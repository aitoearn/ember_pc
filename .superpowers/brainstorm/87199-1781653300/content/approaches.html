测试用例管理 · 阶段一实施架构

前端 React（src/features/test-case-management/）

1a ViewModel(筛选/分组/统计) · 模块树 · 用例表格 · 详情/编辑 Drawer · 批量操作
1b AI 生成面板(文本/文件/URL → 草稿预览 → 批量入库) · 复用 ModelSelector
1c 执行抽屉(用例→自然语言指令) · 结果回写 · 执行历史时间线
↓ AppServerClient.request(method, params)

App Server JSON-RPC（ember-rs/crates/app-server-protocol + app-server）

1a testCase/list·read·save·delete · testCaseModule/list·save·delete
1c testCaseRun/start·list（执行记录）
protocol 类型 · method_names · catalog · processor dispatch · runtime · local_data_source
↓ rusqlite

SQLite 持久化（ember-rs/crates/core · ember.db）

1a 表 test_case_modules · test_cases（workspace_id scoped）+ DAO
1c 表 test_case_runs · test_case_run_steps（执行历史 + 截图路径 + 日志）
方案做法取舍
A · 一把梭 1a/1b/1c 一次性全做完再交付 最贴规则但周期最长、风险集中、不可分段验收
B · 分期纵切（推荐） 1a 后端底座+CRUD/树 UI → 1b AI 生成 → 1c 执行桥+历史；每期独立可交付可验证 每期都打通"前端→App Server→SQLite"一条窄链，早出可用闭环，风险分散
C · 前端先行 先 localStorage 做完整 UI 再迁后端 ❌ 违背你选的 App Server，且仓库禁止生产路径 mock，否决
