# Feature Specification: 确定性可复现测试流与自愈回放

**Feature Branch**: `feature/deterministic-flow-self-healing`

**Created**: 2026-06-17

**Status**: Draft

**Input**: 竞品分析（TestMu AI / Maestro / Maestro Studio / Kobiton / Panto AI）后规划的平台主线。在 Ember 移动端测试工作台「UI 自动测试」既有的 **VLM 自然语言执行**（`useDeviceAiTask` + `ui_agent_start`，非确定性、软断言）之外，新增 **确定性可复现测试流**：把一次 AI 探索 / 自然语言执行（或手动投屏操作）**录制**为结构化、可版本管理的确定性流，之后用 **确定性回放引擎**（优先 selector/UI 树定位，不调大模型）稳定、廉价地复现；当某步因 UI 变动而失配时，**自愈**机制降级到 VLM 重新定位并回写流。来源：`docs/竞品分析-TestMuAI-借鉴需求文档.md` 之 P0-2 + P1-7（及 P1-6 自动等待并入回放可靠性）。

## Clarifications

### Session 2026-06-17（已定，采用推荐默认）

- Q: 确定性流的存储格式？ → A：**自研结构化 JSON 流**（步骤数组 + selector 多策略 + 断言）为首期唯一事实源，并预留 **导出 Maestro YAML / Appium** 的扩展点作为后续互通项（不在首期范围）。（Panto 亦自建专有框架并导出 Maestro/Appium，路线一致）
- Q: 自愈是否随确定性回放紧邻交付？ → A：**是**。自愈（US3）作为确定性回放（US2）的紧邻伴生能力优先排期——这是 Ember 同时拥有 VLM 与确定性流后的独占壁垒，不延后。
- Q: 录制来源？ → A：首期支持两种——(1) 把一次 **VLM 自然语言执行** 的成功轨迹录制成流；(2) 手动投屏操作录制成流。AI 自主爬取生成延后。
- Q: 回放定位优先级？ → A：**selector（resource-id / text / accessibility）优先 → UI 树结构匹配 → VLM 视觉定位兜底**，与既有「UI 树 / Vision / 混合感知」一致。
- Q: 断言模型？ → A：结构化流支持 **硬断言**（selector / 文案精确或包含匹配）为主；保留 VLM **软断言** 作为复杂场景兜底。
- Q: 自愈回写是否自动生效？ → A：自愈成功后默认 **写入「待确认」修订**，由用户确认「预期变更 / 缺陷」后才并入主流；不静默覆盖。
- Q: 首期平台范围？ → A：**Android 优先**（复用现有 ADB / scrcpy / UI 树链路）；iOS/Harmony 在能力矩阵中标注后续。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 把一次 AI/手动执行录制为确定性流 (Priority: P1)

测试人员在「UI 自动测试」工作台用自然语言驱动 AI 完成一段操作（或手动在投屏上点击/输入/滑动），执行成功后点击「保存为流」，系统把该轨迹固化为一条**结构化确定性流**（有序步骤 + 每步定位策略 + 断言），保存到当前工作区、可命名、可重命名、可删除。

**Why this priority**: 录制是整条主线的入口与底座。没有「把探索固化成可复现资产」的能力，确定性回放与自愈都无源可依。它本身即可独立交付（先能录、能存、能看）。

**Independent Test**: 用自然语言执行一段含 3–5 步的操作（如启动应用→进入设置→打开某开关），保存为流，刷新应用后该流仍在、步骤完整、每步带定位信息，即视为通过。

**Acceptance Scenarios**:

1. **Given** 一台在线 Android 设备与一段成功的 VLM 自然语言执行，**When** 用户点击「保存为流」，**Then** 生成一条结构化流，按时间顺序列出每步的操作类型（tap/input/swipe/launch/back/assert 等）与定位信息，且可命名保存。
2. **Given** 用户在投屏上手动完成若干操作，**When** 选择「录制」并停止，**Then** 这些操作被转写为结构化流的有序步骤。
3. **Given** 已保存若干流，**When** 用户重启应用进入工作台，**Then** 流列表仍可加载（按工作区持久化）。
4. **Given** 一条已保存的流，**When** 用户查看详情，**Then** 可见每步的操作、定位策略（如 resource-id/text）、预期断言。

---

### User Story 2 - 确定性回放（不调大模型） (Priority: P1)

测试人员选中一条已录制的流与一台已连接设备，点击「回放」。系统按步骤用 **selector / UI 树** 定位并执行，**默认不调用大模型**；执行时自动等待网络/动画/UI 沉降到稳定再继续；每步用硬断言判定通过/失败；结束产出明确结论（通过/失败/阻塞）与逐步过程（含截图）。同一条流在 UI 不变时多次回放结果一致。

**Why this priority**: 这是相对 VLM 执行的核心增量价值——**稳定、廉价（不烧 token）、可复现的回归**，直接消除 spec 001 承认的软断言不确定性风险。依赖 US1 的流数据。

**Independent Test**: 对 US1 录制的流回放两次，UI 未变时两次结论一致、关键步骤截图齐全、且执行过程未触发大模型调用，即视为通过。

**Acceptance Scenarios**:

1. **Given** 一条已录制流与一台在线 Android 设备，**When** 用户回放，**Then** 各步按 selector/UI 树定位执行，硬断言逐步判定，最终给出通过/失败/阻塞结论。
2. **Given** 同一条流与未变化的应用，**When** 用户连续回放两次，**Then** 两次结论一致（确定性）。
3. **Given** 某步目标元素尚未渲染，**When** 回放到该步，**Then** 系统自动等待至稳定（或超时）再判定，**无需用户手写 sleep**。
4. **Given** 一次回放完成，**When** 用户查看执行记录，**Then** 可见逐步操作、定位结果、断言结论与截图；该次回放**未产生大模型 token 消耗**（除非触发 US3 自愈）。

---

### User Story 3 - 自愈：失配自动降级 VLM 重导并回写 (Priority: P2)

确定性回放过程中，某步 selector 因 UI 变动而失配。系统**自动降级到 VLM 视觉理解**重新定位/重导该步，使流程得以继续；自愈成功后，把修正后的定位策略作为**「待确认修订」**写回该流，并通知用户判断这是「预期内 UI 变更」（接受修订）还是「缺陷」（标记并保留原流）。

**Why this priority**: 自愈解决确定性流最大痛点——UI 一变脚本就失效。它是 Ember **同时拥有 VLM 与确定性流**后的独占差异化（纯确定性工具做不到，纯 VLM 工具无流可修）。但依赖 US1/US2 已稳定，复杂度最高，排最后。

**Independent Test**: 录制一条流，人为改动被测页面（如改按钮文案/位置）后回放，验证：失配步触发 VLM 重导并继续、回放最终给出结论、该流出现一条「待确认修订」、用户接受后再次回放不再触发自愈，即视为通过。

**Acceptance Scenarios**:

1. **Given** 一条流的某步 selector 在新版本应用中失配，**When** 回放到该步，**Then** 系统自动降级 VLM 重新定位并执行，回放不因单步失配而整体中断。
2. **Given** 自愈成功修正了某步定位，**When** 回放结束，**Then** 该流出现一条「待确认修订」，标明原定位与新定位差异，并通知用户。
3. **Given** 用户对修订选择「接受（预期变更）」，**When** 再次回放，**Then** 使用新定位、不再触发该步自愈。
4. **Given** 用户对修订选择「标记为缺陷」，**When** 查看流，**Then** 原流保持不变，且生成一条带证据（截图/差异）的缺陷线索记录。
5. **Given** 自愈降级 VLM 后仍无法完成某步，**When** 超出重试策略，**Then** 该步记为失败/阻塞并保留已产生的过程信息，不静默通过。

---

### Edge Cases

- 录制时设备掉线或截图失败：停止录制并提示，已捕获的步骤可保存为草稿流。
- 回放时目标应用未安装 / 包名变更：明确报错并中止，不产生误判通过。
- 回放时元素持续不可见直至超时：该步记为失败（区分「超时未现」与「断言不符」）。
- selector 多策略全部失配且自愈未启用（US3 未交付或被关闭）：该步记为失败，给出「建议开启自愈或重新录制」提示。
- 自愈频繁触发（同一流多步反复失配）：聚合为「该流疑似大幅过时」提示，引导整条重录而非逐步修补。
- 同一设备同时发起回放与 VLM 自然语言执行：互斥，提示先结束其一。
- 流格式版本升级：旧版本流需可被加载或给出可读的迁移/不兼容提示，不得静默损坏。
- 大量流（数百条）下列表与检索仍保持可用响应速度。

## Requirements *(mandatory)*

### Functional Requirements

#### 流定义与录制（US1）

- **FR-001**: 系统 MUST 定义结构化确定性流的数据格式：有序步骤集合，每步含操作类型（至少 `launch_app` / `tap` / `input_text` / `swipe` / `scroll_until_visible` / `back` / `assert` / `wait`）、定位信息与（可选）断言。
- **FR-002**: 每个定位步骤 MUST 支持**多策略定位**并按优先级存储：selector（resource-id / text / accessibility-id）→ UI 树结构路径 → VLM 视觉锚点（坐标/描述）。
- **FR-003**: 系统 MUST 支持把一次**成功的 VLM 自然语言执行轨迹**转写为结构化流（复用既有 `ui_agent` 步骤事件：thought/action/screenshot/result）。
- **FR-004**: 系统 MUST 支持**手动投屏操作录制**为结构化流（点击/输入/滑动/系统导航映射为对应步骤）。
- **FR-005**: 系统 MUST 支持流的新建、命名、查看、编辑（增删改步骤）、删除，并按当前**工作区隔离持久化**，重启不丢失。
- **FR-006**: 流的步骤断言 MUST 支持**硬断言**（selector/文案的精确或包含匹配）；MAY 保留 VLM 软断言作为兜底断言类型。

#### 确定性回放（US2）

- **FR-007**: 系统 MUST 提供确定性回放引擎，按步骤用 selector → UI 树定位执行，**默认不调用大模型**（除非触发 FR-012 自愈）。
- **FR-008**: 回放 MUST 内置**自动等待**：在判定每步前等待网络/动画/UI 沉降到稳定或达到超时，无需用例显式 `sleep`。
- **FR-009**: 回放 MUST 对每步产出定位结果与断言结论，并最终给出**通过 / 失败 / 阻塞**之一的整体结论，不存在「无结果」静默态。
- **FR-010**: 回放 MUST 持久化一条执行记录（关联流、设备、起止时间、结论、逐步过程与截图），可在历史中回看。
- **FR-011**: 在 UI 未变化的前提下，同一条流多次回放 MUST 给出**一致结论**（确定性）。

#### 自愈（US3）

- **FR-012**: 当某步 selector / UI 树定位失配时，系统 MUST 能**自动降级到 VLM** 重新定位/重导该步，使回放尽可能继续，而非整体中断。
- **FR-013**: 自愈成功修正某步定位后，系统 MUST 将修正生成为一条**「待确认修订」**（含原定位 vs 新定位差异），并通知用户。
- **FR-014**: 用户 MUST 能对待确认修订选择**「接受（预期变更）→ 并入流」**或**「标记为缺陷 → 保留原流并生成缺陷线索（带截图/差异证据）」**。
- **FR-015**: 自愈降级 VLM 后仍无法完成某步时，系统 MUST 将该步记为失败/阻塞并保留过程信息，不静默通过；MUST 有重试与中止策略（避免无限重试）。
- **FR-016**: 自愈触发 MUST 可在流或工作台层级**开关**；关闭时失配步直接判失败并提示。

#### 通用

- **FR-017**: 同一设备上 MUST 互斥「确定性回放」与「VLM 自然语言执行」，避免并发抢占设备通道。
- **FR-018**: 首期 MUST 仅对**在线 Android 设备**开放录制与回放；iOS/Harmony MUST 展示能力矩阵与「后续支持」说明，不可误启动。
- **FR-019**: 流与执行记录 MAY 提供导出能力（结构化 JSON），并 SHOULD 预留导出 **Maestro YAML / Appium** 的扩展点（首期可不实现）。
- **FR-020**: 用户可见文案 MUST 覆盖**简体中文与英文**（zh-CN / en-US，遵循项目 i18n 规则）；错误场景 MUST 给出可读的中文提示。

### Key Entities *(include if feature involves data)*

- **测试流（TestFlow）**: 一条可确定性回放的结构化用例。关键属性：名称、所属工作区、目标应用包名、有序步骤集合、格式版本、来源（VLM 录制 / 手动录制）、创建/更新时间、自愈开关。
- **流步骤（FlowStep）**: 流内一个有序操作单元。含步骤序号、操作类型、**多策略定位（selector / UI 树路径 / VLM 锚点，按优先级）**、可选断言（硬/软）、等待策略。
- **回放记录（FlowRun）**: 一次确定性回放的留痕。含关联流、目标设备、起止时间、整体结论（通过/失败/阻塞）、逐步过程（定位结果、断言结论、截图）、是否触发自愈、token 消耗（确定性回放应为 0，自愈步另计）。
- **自愈修订（HealingRevision）**: 一次自愈产生的待确认变更。含关联流与步骤、原定位、新定位、差异证据（截图）、状态（待确认/已接受/已标记缺陷）、生成时间。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可在一次成功的 AI/手动执行后，**1 分钟内**完成「保存为流」并在列表中看到该流（步骤完整、含定位信息）。
- **SC-002**: 对一条已录制流，UI 未变时**连续两次回放结论 100% 一致**（确定性达成）。
- **SC-003**: 一次确定性回放（未触发自愈）的**大模型 token 消耗为 0**，相对等价的 VLM 自然语言执行**显著更快**（目标：同流程回放耗时 ≤ VLM 执行耗时的 50%）。
- **SC-004**: 人为改动被测页面后回放，自愈机制能让流程**继续完成并给出结论**的比例达到可接受阈值（首期目标：常见文案/位置类变更下 ≥ 70% 可自愈续跑），且每次自愈都生成一条待确认修订。
- **SC-005**: 任意一次回放都能在历史中追溯到记录、逐步过程与截图；失败/阻塞步均有可读原因。
- **SC-006**: 非 Android 平台**0 次**误启动录制/回放。
- **SC-007**: 所有用户可见文案在 zh-CN / en-US 下均有对应翻译，无兜底缺漏。

## Assumptions

- **与现有能力的关系**：本功能是 UI 自动测试「VLM 自然语言执行（已可用）」的**互补增量**，不替代它；VLM 执行继续作为「快速探索 / 零门槛编写 / 自愈兜底」的引擎。
- **复用既有底座**：设备连接与截图复用现有 ADB / scrcpy；定位复用现有 UI 树 / Vision / 混合感知；模型/Provider 复用现有配置（`apiKeyProvider`）；持久化复用应用本地数据库并按工作区隔离。
- **流格式事实源**：首期以**自研结构化 JSON** 为事实源；Maestro YAML / Appium 导出为后续互通项（FR-019 预留扩展点），不在首期硬性范围。
- **断言语义**：硬断言为主、软断言兜底；硬断言提供确定性，软断言保留对复杂/动态界面的容错，二者在同一流中可混用。
- **自愈边界**：自愈用于「定位失配的修复」，不用于「业务断言失败的掩盖」——断言不符仍判失败；自愈仅尝试让"该点哪里/输入到哪里"重新对齐。
- **平台范围分期**：P1/P2 仅 Android 真实可用；iOS/Harmony 录制回放延后。
- **与 spec 001 的衔接**：spec 001 的结构化用例步骤可作为流格式的设计起点与导入来源；spec 001 的单条执行（P3）可逐步迁移到本功能的确定性回放 + VLM 兜底。
- **与 spec 002 的衔接**：回放期间可联动性能采集，并把性能片段并入回放记录与统一报告（属后续报告需求 P1-1，不在本 spec 强制范围）。
- **交付分期纵切**：US1 录制底座 → US2 确定性回放 → US3 自愈，每期可独立交付与验证。
- **超出本 spec 范围**：Studio 式点选生成步骤（编写效率增强）、统一测试报告/趋势/门禁、CLI/CI、PM 集成、自主爬取、视觉回归、多设备并行均为独立需求（见 `docs/竞品分析-TestMuAI-借鉴需求文档.md`），不在本功能包内。
