# Feature Specification: 测试用例管理

**Feature Branch**: `feature/test-case-management`

**Created**: 2026-06-17

**Status**: Draft

**Input**: User description: "在 ember 桌面端新增测试用例管理页面，支持用例的结构化管理、AI 辅助生成、以及与现有设备自动化（VLM 智能体）联动的执行能力。"

## Clarifications

### Session 2026-06-17

- Q: 业务用例编号 caseId 的唯一性策略？ → A: 工作区内强制唯一，保存时校验冲突（冲突则拒绝保存并提示）。
- Q: 删除仍含子模块或用例的模块时如何处理？ → A: 拒绝删除非空模块，提示用户先移动或清空子项。
- Q: 本期执行能力的范围？ → A: 仅单条用例执行；批量/整模块执行、多设备并发延后到后续阶段。
- Q: AI 生成面板给用户多少生成控制项？ → A: 轻量可选——期望数量 + 关注的用例类型；留空则由模型自行决定。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 用例结构化管理底座 (Priority: P1)

测试人员打开「测试用例管理」页面，按模块树（项目→模块→子模块）组织用例；新建、编辑、删除用例与模块；用关键词与多维条件（模块/优先级/类型/状态/来源/执行结果）筛选；批量调整状态、优先级、模块或批量删除。所有数据按当前工作区隔离并持久化，重启后仍在。

**Why this priority**: 这是整个功能的底座。没有用例的录入、组织、检索能力，AI 生成与执行都无处落地。它本身即可独立交付，作为「能管理用例」的最小可用产品。

**Independent Test**: 仅实现本故事即可完整验证——新建一个模块和若干用例、编辑字段、按条件筛选出目标子集、批量改状态、删除，刷新页面后数据仍正确，即视为通过。

**Acceptance Scenarios**:

1. **Given** 空的用例库，**When** 用户新建模块「登录」并在其下新建用例「手机号登录-正常」，**Then** 模块树出现「登录」节点，用例列表显示该用例且可点开查看结构化步骤与预期。
2. **Given** 用例库已有数十条用例，**When** 用户输入关键词并叠加「优先级=P0、状态=已评审」筛选，**Then** 列表只显示同时满足关键词与全部条件的用例。
3. **Given** 列表中多选了 5 条用例，**When** 用户执行批量「设为已评审」，**Then** 这 5 条状态全部变为「已评审」且立即反映在列表与统计上。
4. **Given** 用户编辑了某用例并保存，**When** 重新打开应用进入该页面，**Then** 改动后的内容仍然存在（持久化成功）。

---

### User Story 2 - AI 辅助生成用例 (Priority: P2)

测试人员在生成面板提供需求来源（粘贴文本、本地 md/txt、本地 docx/pdf、或在线文档 URL），选择模型，触发生成；系统输出一批用例草稿；用户在草稿列表中逐条勾选、编辑、删除，确认后批量入库（来源标记为「AI生成」、状态默认「草稿」）。

**Why this priority**: 在底座可用之后，AI 生成显著降低用例编写成本，是核心增值点。依赖 P1 的入库能力，因此排在其后。

**Independent Test**: 在已具备用例管理能力的前提下，粘贴一段需求文本生成草稿，编辑其中一条、删除一条，批量入库剩余草稿，确认入库后的用例出现在列表中且来源为「AI生成」，即视为通过。

**Acceptance Scenarios**:

1. **Given** 用户粘贴了一段功能需求文本并选好模型，**When** 点击生成，**Then** 面板展示一组结构化用例草稿（含标题、步骤、预期）。
2. **Given** 一批草稿已生成，**When** 用户取消勾选其中两条并修改一条标题，**Then** 仅勾选中的草稿（含修改）进入入库范围。
3. **Given** 用户确认批量入库，**When** 入库完成，**Then** 这些用例出现在对应模块下，来源为「AI生成」、状态为「草稿」。
4. **Given** 用户提供了一个本地 docx 文件，**When** 触发生成，**Then** 系统能读取其正文内容作为生成输入。

---

### User Story 3 - 用例执行与结果追溯 (Priority: P3)

测试人员选中一条用例与一台已连接设备，触发执行；系统把结构化用例转成自然语言指令交给设备智能体执行，并据「通过条件」自评判定结果（通过/失败/阻塞）；执行结束后结果回写到用例，并生成一条带过程观察与截图的执行记录，可在历史中回看。

**Why this priority**: 执行与追溯把用例从「文档」变成「可运行的验证」，是最终价值闭环，但依赖 P1 的用例数据与现有设备自动化能力，复杂度最高，排在最后。

**Independent Test**: 在已有用例和一台可用设备的前提下，对一条用例发起执行，得到通过/失败/阻塞之一的结果并回写到用例，执行历史中出现本次记录与过程截图，即视为通过。

**Acceptance Scenarios**:

1. **Given** 一条含前置条件、步骤与预期的用例和一台已连接设备，**When** 用户发起执行，**Then** 设备智能体按指令操作并在结束时给出通过/失败/阻塞结论。
2. **Given** 一次执行已完成，**When** 用户查看该用例，**Then** 用例的「执行结果」字段更新为本次结论。
3. **Given** 同一用例被执行过多次，**When** 用户打开执行历史，**Then** 可看到按时间排列的多条记录，每条含结论、过程观察与截图。

---

### Edge Cases

- 删除非空模块被拒绝，提示先移动或清空子项（见 FR-001a）。
- AI 生成返回的内容无法解析为结构化用例（格式错乱/为空）时，面板需给出可读的失败提示而非静默丢弃。
- 提供的 URL 无法访问或 docx/pdf 解析失败时，需明确报错并允许用户改用粘贴正文。
- 执行过程中设备掉线、智能体异常或超时，结果应记为「阻塞」并保留已产生的过程信息。
- 业务用例编号在工作区内重复时，保存被拒绝并提示用户改用未占用的编号（见 FR-002a）。
- 大量用例（数千条）下筛选、分组、统计仍需保持可用的响应速度。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 允许用户以树形模块（项目→模块→子模块）组织用例，并支持模块的新建、重命名、移动排序与删除。
- **FR-001a**: 系统 MUST 拒绝删除仍含子模块或用例的模块，并提示用户先移动或清空其子项；仅空模块可被删除。
- **FR-002**: 系统 MUST 支持用例的新建、查看、编辑、删除，用例包含标题、业务编号、所属模块、优先级、类型、状态、来源、前置条件、结构化步骤（操作+预期）、标签、备注、执行结果等字段。
- **FR-002a**: 业务用例编号（caseId）MUST 在同一工作区内唯一；保存用例时系统 MUST 校验编号冲突，冲突则拒绝保存并给出可读提示。
- **FR-003**: 系统 MUST 支持按关键词（匹配标题/编号/步骤/标签）与多维条件（模块、优先级、类型、状态、来源、执行结果）组合筛选，并展示与筛选结果一致的统计摘要。
- **FR-004**: 系统 MUST 支持对多选用例的批量操作（至少：批量改状态、改优先级、改模块、删除）。
- **FR-005**: 系统 MUST 将用例与模块数据按当前工作区隔离持久化，应用重启后数据不丢失。
- **FR-006**: 用户 MUST 能从多种来源为 AI 生成提供输入：纯文本粘贴、本地 md/txt 文件、本地 docx/pdf 文件、在线文档 URL。
- **FR-007**: 系统 MUST 允许用户在生成前选择使用的模型（复用 ember 现有的模型/供应商配置）。
- **FR-008**: 系统 MUST 在生成后以草稿列表形式呈现结果，支持逐条勾选、编辑、删除，再批量入库；入库用例来源标记为「AI生成」、初始状态为「草稿」。
- **FR-008a**: 生成面板 MUST 提供可选的轻量生成控制（期望生成数量、关注的用例类型）；用户留空时由模型自行决定，不强制填写。
- **FR-009**: 系统 MUST 支持对单条用例选择一台已连接设备发起执行，并把结构化用例转换为包含前置条件、操作步骤与「通过条件」的自然语言指令交给设备智能体执行。
- **FR-010**: 系统 MUST 在执行结束后产出通过/失败/阻塞之一的结论，回写到用例的执行结果字段，并保存一条含过程观察与截图的执行记录。
- **FR-011**: 用户 MUST 能查看某用例的历史执行记录列表及每次执行的详情（结论、过程、截图）。
- **FR-012**: 页面 MUST 接入应用的主导航（侧边栏入口），用户可见文案 MUST 覆盖五种语言（zh-CN / zh-TW / en-US / ja-JP / ko-KR）。
- **FR-013**: 系统在 AI 生成解析失败、文件/URL 读取失败、执行异常等错误场景下 MUST 向用户给出可读的中文错误提示。

### Key Entities *(include if feature involves data)*

- **测试用例（TestCase）**: 一条可被评审与执行的验证单元。关键属性：业务编号、标题、所属模块、优先级、类型、生命周期状态、来源、前置条件、结构化步骤集合、标签、最近执行结果、备注、创建/更新时间。
- **测试步骤（TestCaseStep）**: 用例内的一个有序操作单元，含步骤序号、操作描述、该步预期结果。
- **测试模块（TestCaseModule）**: 树形组织节点，含名称、父节点、同级排序；用例归属于某模块。
- **执行记录（TestCaseRun）**: 一次执行的留痕，含关联用例、目标设备、生成的指令、结论（通过/失败/阻塞）、摘要、起止时间，以及若干过程步骤（观察文本、截图）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可在 1 分钟内完成「新建模块 + 在其下新建一条含 3 步的用例并保存」的完整操作。
- **SC-002**: 在含数千条用例的库中，任意组合筛选返回结果的可感知等待时间不超过 1 秒。
- **SC-003**: 从一段需求文本到「生成草稿并批量入库至少 1 条用例」的端到端流程，用户无需手写步骤即可完成。
- **SC-004**: 对一条用例发起执行后，用户最终一定能得到通过/失败/阻塞之一的明确结论，不存在「无结果」的静默态。
- **SC-005**: 所有用户可见文案在五种语言下均有对应翻译，无中文/英文兜底缺漏。
- **SC-006**: 任意一次执行均可在历史中追溯到对应记录与过程截图。

## Assumptions

- **执行采用「软断言」模型**：现有设备自动化是自然语言驱动的视觉智能体（VLM ReAct），无原生结构化步骤执行与硬断言；用例的「预期结果」作为「通过条件」嵌入自然语言指令，由智能体自评。判定具备可解释性但非确定性，不保证逐字符精确比对。该取舍已与需求方确认接受。
- **输入来源范围**：AI 生成本期支持纯文本、md/txt、docx/pdf、URL；不含图片截图、不含知识库作为输入源。
- **导入/导出（Excel/JSON）本期不做**，延后到后续阶段。
- **复用既有能力**：模型/供应商选择复用 ember 现有配置；设备执行复用现有设备自动化能力；不为本功能新建独立的模型或设备体系。
- **单工作区视角**：用例与模块按当前工作区隔离；跨工作区共享/迁移本期不做。
- **评审流程从简**：保留用例「状态」字段（草稿/待评审/已评审/已废弃），但不实现多人评审流转工作流引擎。
- **执行范围本期仅单条用例**：批量执行、整模块跑一轮、多设备并发执行均延后到后续阶段。
- **交付方式为分期纵切**：P1 底座 → P2 AI 生成 → P3 执行与追溯，每期可独立交付与验证。
